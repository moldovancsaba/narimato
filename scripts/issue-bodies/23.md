## Objective
Make database setup scripts use the same Organization (and Card) schemas as runtime `lib/models/*`, and document what operators should run for the chosen tenancy model (ADR #17).

## Unified Context
`scripts/setup-databases.js` defines inline Organization schema with `displayName`, `databaseName`, `settings`. `scripts/init-default-org.js` uses `name` + `databaseName` + Card with `body` field. Runtime `lib/models/Organization.js` has `name`, `slug`, `description` only—no `databaseName`.

Operators running scripts get different documents than the app expects.

## Problem
Setup scripts seed structures the API never reads, while missing fields the UI uses. This wastes onboarding time and implies per-org DB switching that does not happen.

## Goal
Scripts import Mongoose models from `lib/models/` (or shared schema module), produce org records the app can list via `/api/organizations`, and include README operator instructions aligned with ADR #17.

## Scope
### In Scope
- Refactor `setup-databases.js`, `init-master-db.js`, `init-default-org.js` to require shared models
- Remove inline duplicate schemas where possible
- Document in unified doc or `scripts/README.md`: when to run each script, required env vars
- If ADR #17 Path B: scripts only seed master metadata; clarify `MONGODB_URI` targets app DB

### Out of Scope
- Implementing per-org DB routing (ADR #17 Path A is separate implementation issue)
- Data migration for existing Atlas documents

## Execution Prompt
After #17 ADR, update scripts. Replace inline schemas with `require('../lib/models/Organization')`. Fix Card seed shape to `title`/`description`/`organizationId`. Test script against dev Atlas or document dry-run.

## Constraints
- Scripts must load `.env.local` consistently (`dotenv` path)
- Do not break `seed-superadmin.js` if it depends on old fields—update together

## Acceptance Checks
- [ ] Scripts create org readable by `GET /api/organizations`
- [ ] No inline Organization schema diverging from `lib/models/Organization.js`
- [ ] Operator doc states single-DB vs multi-DB clearly per ADR
- [ ] Card seeds validate against `Card.js` schema

- **Parent program:** #11

- Existing deployments have old-shaped org documents—note manual migration in PR
