## Objective
Define and enforce a single `mode` string contract in play results responses across all engines, or document the allowed per-engine values if normalization is deferred.

## Unified Context
`POST /api/v1/play/start` accepts snake_case modes (`vote_only`, `swipe_more`, …). Results payloads from engines return inconsistent strings: e.g. `swipe-only` vs `swipe_only`, `vote-more` vs `vote_more`. API_REFERENCE examples mix both. Clients parsing results for analytics or UI labels will break silently.

## Problem
Inconsistent enums are a stable API bug that documentation alone cannot fix if clients already depend on one format.

## Goal
Either:
- **Option A (preferred):** Normalize all engines to snake_case matching start API, update API_REFERENCE
- **Option B:** Publish explicit enum table per engine in API_REFERENCE until normalization ships

## Scope
### In Scope
- Inventory `mode` values returned by: `VoteOnlyService`, `SwipeOnlyEngine`, `SwipeMoreEngine`, `VoteMoreEngine`, `RankOnlyEngine`, `RankMoreEngine`
- Choose canonical format (recommend **snake_case**)
- Code changes in engine result builders OR documentation table
- Update `pages/results.js` if it branches on mode strings

### Out of Scope
- Renaming URL query params in `pages/play.js` (`vote-only` kebab-case in URLs is fine)

## Execution Prompt
Grep `lib/services` for `mode:` in return objects. Build spreadsheet → implement normalization in one PR or document in API_REFERENCE with “Breaking in v7.3” note if clients exist.

## Constraints
- If changing strings, bump API version or document breaking change in RELEASE_NOTES
- GA4 events in `play.js` should use same canonical mode string

## Acceptance Checks
- [ ] API_REFERENCE results section lists one enum format
- [ ] All engines return canonical format OR doc table is complete and tested
- [ ] `npm run build` passes
- [ ] Manual smoke: start swipe_only + vote_only, fetch results, verify `mode` field
