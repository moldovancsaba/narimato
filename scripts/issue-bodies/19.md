## Objective
Publish authoritative documentation for legacy `pages/api/play/*` routes that remain in production use, including migration guidance to `/api/v1/play/*`.

## Unified Context
README states all play modes use the unified API. `pages/play.js` contradicts this:
- Unified modes use `/api/v1/play/start`, `input`, `next`
- Classic flow (no mode / unset) uses `/api/play/start`, `swipe`, `vote`
- Hierarchical helpers use `/api/play/hierarchical-status`, `current`, etc.

Removing these routes without a client migration would break play sessions in the field.

## Problem
Undocumented legacy routes are “shadow APIs”—only discoverable by reading 1400+ lines of `play.js`. Agents and integrators will not find them.

## Goal
`docs/DEPRECATED_API.md` (or API_REFERENCE deprecated chapter) listing each legacy route, body, caller in `play.js`, and v1 equivalent if any.

## Scope
### In Scope
- Document at minimum: `start`, `swipe`, `vote`, `current`, `results`, `hierarchical-status`, `expand`, `cleanup`, `rankChildren`, `merge`
- For each: HTTP method, handler file, engine (`DecisionTreeEngine`, etc.), when `play.js` calls it
- Migration notes: which query modes still hit legacy
- Mark status: **Deprecated — do not use for new integrations**

### Out of Scope
- Deleting legacy routes (follow-on issue after client migration)
- Refactoring `play.js` in this issue

## Execution Prompt
Grep `pages/play.js` for `/api/play/`. For each path, open handler under `pages/api/play/` and document contract. Add table “Unified replacement” where 1:1 mapping exists.

## Constraints
- Do not document routes that always 410 unless explaining removal (`/api/system/page-passwords`)
