## Objective
Make API version negotiation consistent across all unified play endpoints by applying `withApiVersion` / `resolveVersioned` to `start` and `input`, or formally document why they are exempt.

## Unified Context
`docs/API_REFERENCE.md` describes version negotiation via `Accept: application/vnd.narimato.vN+json` and `X-API-Selected-Version` response headers. `pages/api/v1/play/[playId]/next.js` and `results.js` use `lib/middleware/apiVersion.js`; **`start.js` and `input.js` do not**, so clients cannot rely on uniform versioning behavior.

## Problem
Partial versioning is worse than none—clients assume headers on all v1 routes and mis-handle start/input responses.

## Goal
Either full parity (wrap handlers, emit `X-API-Selected-Version`) or explicit “start/input are unversioned v1-only” in API_REFERENCE with rationale.

## Scope
### In Scope
- Audit `apiVersion.js` usage across `pages/api/v1/play/*`
- Wrap `start.js` and `input.js` if low-risk
- Add versioned payload maps if v2 shapes already exist for next/results
- Update API_REFERENCE negotiation section

### Out of Scope
- Breaking v2 API design beyond existing stubs
- Versioning legacy `/api/play/*`

## Execution Prompt
Read `withApiVersion` implementation. Mirror pattern from `next.js`. Test with curl sending `Accept: application/vnd.narimato.v1+json` and v2 if defined. Update docs.

## Constraints
- Default remains v1 when no header
- Rate limiting unchanged

## Acceptance Checks
- [ ] start + input return `X-API-Selected-Version` OR docs state exemption
- [ ] API_REFERENCE matches behavior
- [ ] Existing clients without Accept header unchanged
- [ ] `npm run build` passes
