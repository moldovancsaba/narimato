## Objective
Add client-side vote debouncing in `pages/play.js` so rapid taps do not send duplicate vote requests, matching the vote-integrity story documented in WARP and unified spec.

## Unified Context
`VoteOnlyService` dedupes identical votes within **2 seconds** server-side. `WARP.md` claims **100ms client debounce**—no such logic exists in `handleVote` (only unrelated 100ms resize debounce). Mobile users can double-tap comparison cards and trigger parallel `/api/v1/play/.../input` calls.

## Problem
Relying only on server dedupe still wastes API quota, causes racey UI transitions, and makes analytics (`vote_cast` GA4) noisy.

## Goal
100ms debounce (or documented alternative) on vote actions before fetch, plus updated docs if timing changes.

## Scope
### In Scope
- Debounce `handleVote` in `pages/play.js` for v1 vote paths
- Disable vote buttons briefly while request in flight (optional but recommended)
- Update WARP + unified doc vote-integrity subsection with accurate client+server behavior
- Verify no regression for rank-only / vote-more / rank-more branches using same handler

### Out of Scope
- Swipe debounce (separate gesture concern)
- Changing server 2s dedupe window

## Execution Prompt
Implement `useRef` last-vote timestamp or lodash-free debounce helper. Manual test: double-click winner quickly—only one network call. Document in PR.

## Constraints
- Must not block legitimate rapid votes after comparison updates (debounce per tap burst, not per session)
- Keep accessibility: keyboard vote paths included

## Acceptance Checks

- [ ] WARP vote-integrity section matches implementation
- [ ] `npm run build` passes
