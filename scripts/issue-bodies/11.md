## Objective
Establish a single, accurate technical story for `{narimato}` so contributors, operators, and AI agents can rely on documentation that matches the running Next.js + MongoDB codebase.

## Unified Context
A May 2026 doc-vs-code audit found multiple competing “sources of truth” (`README.md`, `WARP.md`, `ARCHITECTURE.md`, `docs/API_REFERENCE.md`, `narimato_unified_documentation.md`) describing different stacks, APIs, data models, and product features. The application at v7.1.0 works, but documentation still advertises capabilities that were never implemented (multi-tenant DB routing, org theming, dark mode, optimistic locking) and contradicts itself on vote-only and ranking algorithms.

The product only converges when **one canonical spec** matches `lib/`, `pages/api/`, and `pages/play.js`.

## Problem
If contradictory docs remain first-class, every change will:
- send implementers to wrong API paths (`/api/v1/cards` vs `/api/cards`)
- mis-set expectations on ranking (binary search vs `VoteOnlyService`)
- block onboarding (WARP forbids `uuid` while models require it)
- recreate the same audit failures on the next release

## Goal
Deliver a coordinated documentation and architecture-alignment program tracked on [project #33](https://github.com/users/moldovancsaba/projects/33), shipping as **v7.2.0** when Phase 0 completes.

## Scope
### In Scope
- Phase 0: canonical spec, API reference, WARP/README repair, vote-only ADR
- Phase 1: multi-tenant ADR, ranking/legacy API truth, model inventory
- Phase 2: repo hygiene (`* 2.js`), script/schema alignment, small API/client fixes
- Milestone **Doc & Architecture Alignment** and linked deliverable issues

### Out of Scope
- New play modes or ranking algorithm rewrites (unless an ADR explicitly chooses implementation over documentation)
- Full removal of legacy `/api/play/*` in this program (document first; removal is a follow-on)
- Visual redesign, org theming implementation, or dark-mode build-out (unless separately scheduled)

## Execution Prompt
Run this program as a sequence of focused issues, not one giant PR. Resolve **product ADRs first** (#14 vote-only, #17 multi-tenant), then lock the canonical spec (#12), then fix API reference and agent docs (#13, #15), then README (#16). Phase 1 and 2 follow once Phase 0 merges.

**Plan:** `docs/DOC_ALIGNMENT_IMPLEMENTATION_PLAN.md`
