## Objective
Document how personal and global rankings are computed in production paths so engineers do not implement or describe the wrong algorithm.

## Unified Context
Marketing and docs emphasize **binary search O(log n)** placement. The **v1 unified API** vote modes (`vote_only`, `vote_more`, `rank_only` vote segment, etc.) use `VoteOnlyService` with challenger/opponent flow and `rankedDeck` / `unrankedDeck`—not `BinarySearchEngine`.

`BinarySearchEngine` in `lib/services/decisionTree/BinarySearchEngine.js` is used inside `DecisionTreeService`, which is **not** on the `PlayDispatcher` path for unified modes. Legacy `pages/api/play/vote.js` uses `DecisionTreeEngine` (separate implementation). **Global** rankings use ELO on `Card.globalScore` via `lib/utils/ranking.js`—there is no `GlobalRanking` collection.

## Problem
Misdocumented algorithms cause wrong optimizations, bad UX expectations (“log n comparisons”), and incorrect academic/marketing claims.

## Goal
Unified doc + README + API_REFERENCE sections that name:
1. Personal ranking per mode (engine class, model, comparison strategy)
2. Global ranking (ELO fields on Card, when updated)
3. Legacy vs v1 path diagram
4. Fate of `BinarySearchEngine` (wire in, legacy-only, or remove)
